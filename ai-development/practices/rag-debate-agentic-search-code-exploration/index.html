    RAG不要論争を整理：Claude Codeが「ベクトルDB型RAG」を捨てた理由と、コード検索の現実解¶

    コード検索の話に限れば、Claude Codeの判断はかなり明確である。 捨てたのはRAG全体ではない。事前インデックスに依存するベクトルDB型RAGである。 問題は単純な精度不足ではない。コード探索では、運用と参照確実性の負担が重かった。 具体的には、staleness・運用負債・セキュリティ・参照の確実性である。

    ただし、この判断をそのまま「RAGは死んだ」と一般化すると、論点を見誤る。 概念検索・巨大repo・非コード知識では、semantic index（埋め込み索引）がなお有効だからである。

    本記事の問いは二つである。なぜClaude CodeはベクトルDB型RAGを捨てたのか。さらに、その判断を自分のコード検索にどこまで一般化してよいのか。

    対象 / ポイント

    対象: Claude Code / Cursor などのAIコーディングツールで、コード検索の設計を見直したい開発者・アーキテクト

    ポイント:

    • Claude Codeが捨てたのは RAG一般 ではなく ベクトルDBに依存した事前インデックス型の検索
    • コード探索では Agentic Searchが勝ちやすい理由 があり、反対に semantic index が残る場面もある
    • 判断は思想ではなく TTFRF / Tokens-per-task / Staleness incidents の3指標で決める

    1. 発火点：Claude Code開発者の一次発言（"RAG→Agentic"）¶

    発火点になったのは、Claude Code開発者の Boris Cherny 氏の投稿である。要旨は次の通りだ。

    • 初期のClaude Codeは RAG + ローカルベクトルDB を使っていた
    • しかしすぐに agentic search の方が概ね良い と分かった
    • よりシンプルで、security / privacy / staleness / reliability の問題が小さい 5

    注意

    ここで対比されている「RAG」は、現場でよく指される"embeddings + vector DB の事前インデックス"、つまりベクトルDB型のRAGとして読むのが自然である。

    2. まず用語を固定する：ベクトルDB型RAGとAgentic Search¶

    Claude Codeが採用しなかったのは、コード探索でよく使われる「embeddings + vector DB + 事前インデックス」である。 この記事では実務上の便宜として、これを ベクトルDB型RAG と呼ぶ。 ここを分けないと、この先の議論は最初から噛み合わない。

    区分定義具体例
    ベクトルDB型RAG（本記事で主に扱う「RAG」）チャンク化 → embeddings → vector DB → semantic retrievalCursorの索引、Pinecone / Weaviate
    Agentic SearchLLMが grep/ls/read/edit を繰り返し呼び、探索をループして必要情報を集めるClaude Code の探索、サブエージェントによる並列検索

    用語の切り方には別流儀もある。学術的には、「検索して生成に使う」ならAgenticも広い意味でRAGに含められる、という整理もある。 ただ、現場の実装や運用を判断するうえでは、"インデックス型か、ライブ探索型か"を分けた方が話が早い。

    3. なぜ"コード探索"ではAgentic Searchが勝ちやすいのか¶

    理由は一つではない。コード探索で欲しい性質と、ベクトルDB型RAGが抱える制約が 噛み合いにくいからである。 特に効くのは、鮮度・参照確定・運用負債・データ境界の4点である。

    3.1 staleness：インデックスは、速いが古くなる¶

    コードは日々変化する。インデックスは必ず古くなる。 これを正しく回すには、更新・権限境界・監査を含む追加設計が増える。

    一方のCursorは、まさにこの"運用負債"を正面から解く方向に進んでいる。 「組織内のクローンは高い類似度を持つ」前提で、チームメンバーの既存インデックスを安全に再利用し、Time-to-first-queryを大幅に短縮する設計を公開した。 1

    要するに、stalenessや運用が重いからRAGが終わりなのではない。RAG側も"運用に耐える設計競争"に入っているのが実態である。

    3.2 reliability：コードは"似てる"より"参照が正しい"¶

    バグ修正やリファクタでは、「似ているコード片」よりも正確な情報が重要である。

    • 正確なシンボル名
    • import/参照元
    • 呼び出し箇所
    • ファイルパス

    Agentic Searchは、grepやファイル読みによって証拠を積み上げ、参照関係を確定しやすい。 ここで、semantic retrieval が見当違いの候補を返したときの手戻りを減らせる。

    3.3 simplicity：稼働部品が増えない＝壊れにくい¶

    インデックス生成・保存・同期・権限・漏洩対策は、プロダクトとして見れば"もう1つのシステム"である。 Agentic Searchは既存のツール（rg/ls/cat等）とファイル実体を使うため、障害点が増えにくい。

    3.4 security / privacy：データの行き先を最小化しやすい¶

    クラウドembeddingや外部DBは、設計を誤ると情報流出の経路になり得る。 ローカル探索中心の構成は「データを外に出さない」と説明しやすく、企業導入では通しやすい。

    4. ただしAgentic Searchにも"刺される弱点"がある¶

    どちらか片側だけを持ち上げると、この議論はすぐに崩れる。 だから先に弱点も出しておく。Agentic Searchにも、コスト上振れと初動の弱さという明確な負け筋がある。

    4.1 トークン/時間が膨らむ（探索ループ・試行錯誤の連鎖）¶

    実際にClaude CodeのIssueでは、「レビューや検索でトークンを燃やすので、コードベースの索引が欲しい」という要望が出ている。2

    要するに、Agenticは"最悪ケースのコスト上振れ"を抱える。ここはRAG側の反論として最も強い。

    4.2 概念検索の初動が弱い¶

    「名前が分からない」「仕様用語と実装用語がずれている」局面では、semantic index がナビとして効く。 Cursorも、大規模コードベースの初動探索では "semantic search が性能を左右する" と位置づけ、インデックスの高速化を最重要テーマの一つとして扱っている。1

    4.3 ローカル探索でも"権限/監査"は消えない¶

    ローカルであっても、「どのディレクトリを読ませるか」「秘密情報をどう除外するか」「どの操作を許可するか」という設計は必要である。 Claude Codeはモデル設定や挙動を複数経路で切り替えられるため、運用ポリシー側にも工夫の余地がある。3

    5. Claude Codeの判断を、どこまで一般化してよいか¶

    一般化の鍵は、ツール名ではなく条件で見ることである。 見るべき条件は3つある。

    • シンボル名やファイルパスが分かっているか
    • 概念検索が先に必要か
    • 索引の鮮度がどれだけ重要か

    Claude Codeの結論を移植できるのは、この条件が揃う現場に限られる。

    条件どちらが有利か理由
    シンボル名やファイルパスがある程度分かっているAgentic Searchgrep / read で参照関係を確定しやすい
    名前が分からず、仕様語と実装語がずれているSemantic Index概念検索の初動を補助しやすい
    変更が頻繁で、古い索引に引っ張られたくないAgentic Searchライブ探索の方が鮮度を保ちやすい
    巨大repoを横断して候補を絞り込みたいHybrid初動は semantic、確定は agentic が安定しやすい
    コード以外の社内知識も混ぜて答えたいベクトルDB型RAG / 別系統の検索非コード知識は事前索引の恩恵が残る

    つまり、Claude Codeの判断を一般化してよいのは、コード探索で正確な参照関係と鮮度が勝敗を決める場面である。 この条件を外れるなら、ベクトルDB型RAGやハイブリッドの方が自然である。

    6. 2026年の現実解：二項対立ではなく"ハイブリッド＋context engineering"¶

    コード探索ではAgenticを基軸にし、概念検索や巨大repoの初動だけ semantic index を足す。 これが、2026年時点で最も失敗しにくい構成である。 最近の整理として分かりやすいのが、「RAGは死んでない。ただ"context engineering"が主戦場になった」という見立てである。4

    ここで言う context engineering とは、検索方式の優劣だけではなく、必要な文脈をどう作ってモデルへ渡すかという設計である。 これをコード探索に翻訳すると、次のようになる。

    6.1 典型的な勝ち筋アーキテクチャ¶

    graph LR
    A[コード探索タスク] --> B{探索タイプ?}
    B -->|シンボル名/パス既知| C[Agentic Search<br/>grep / ls / read]
    B -->|概念/仕様用語| D[Semantic Index<br/>embeddings検索]
    C --> E[探索ログ圧縮<br/>要約/スケッチ]
    D --> E
    E --> F[LLM生成<br/>コード修正/回答]
    F --> G[権限/除外ガード]
    レイヤー役割具体例
    基軸：Agentic Search正確性・鮮度・参照確定grep / ls / read のループ
    補助：Semantic Index概念検索、巨大repoのナビ、横断探索embeddings + vector DB
    圧縮：要約/スケッチ探索ログを短く保ち、再読を減らすコンテキスト要約
    ガード：権限/許可リスト漏洩と事故を潰す.claude/settings.json 等

    7. 迷ったらこれを測る：3つの実測KPI¶

    「閾値（LOC何万行）」は根拠が薄い

    「〇万行以上ならRAG」のような閾値は再現性が低く、反論されやすい。代わりに測れるKPIを置く方が強い。

    閾値論より、同じ2〜3タスクを実測した方が早い。 見る指標は3つだけでよい。TTFRF（当たりファイルに最初に届くまでの時間）、 Tokens-per-task、Staleness incidents である。

    KPI何が分かる目安の読み方
    Time-to-first-relevant-file（TTFRF）当たりファイルに最短で辿り着けるか概念探索が多いならRAG/semanticが効く
    Tokens-per-taskコスト上振れのしやすさAgenticが暴れるならhybrid検討
    Staleness incidents古い文脈で手戻りした回数多いなら"ライブ探索"が有利

    おすすめ手順¶

    1. タスク選定 同じ2〜3タスクを用意する。対象は、バグ修正、リファクタ、新機能追加などでよい。

    2. A/B実行

    3. (A) Agenticのみ（Claude Code標準）

    4. (B) Semantic indexあり（Cursor等のRAG寄りツール）

    5. 比較 上記3指標を比較し、"あなたの現場"の最適解を見つける

    8. 結論（コード検索限定）¶

    コード検索で争点になるのは「RAGかAgenticか」ではなく、結局この3つである。

    1. 鮮度（staleness）
    2. 当たりへの到達速度（TTFRF）
    3. 総コスト（トークン/時間/運用）

    結論は単純である。 多くの現場では、まずAgentic Searchを基軸に置く。semantic index は、 概念検索・巨大repo・横断探索にだけ限定して足す。 この順番で考えると、議論は宗教戦争ではなく設計判断として収束する。

    争点は方式名ではない。どこでライブに読み、どこで先に圧縮しておくかである。 まずAgenticで組み、必要な場所にだけ索引を足す。この順番が最も現実的である。
