    GitHub Copilot Reasoning Levels Guide — Low/Medium/High/x-High Explained¶

    For / Key Points

    For: GitHub Copilot (Pro/Pro+/Enterprise) users who want to understand what the reasoning level selector actually controls

    Key Points:

    • Reasoning levels control how much the model "thinks" — same model, different thinking token budgets
    • Premium request consumption does not change with reasoning level — High/x-High costs the same as Low
    • Default to High. Low is only useful for boilerplate generation and simple syntax questions

    What Are Reasoning Levels?¶

    VS Code 1.113 (March 25, 2026) introduced the Thinking Effort selector — a control that adjusts how deeply the model reasons before generating a response.

    The model picker now shows a submenu with Low / Medium / High. For GPT-5.4 and GPT-5.3-Codex, x-High is available by manually setting "xhigh" in settings.json.

    The key point: changing the reasoning level does not change the model itself. The same GPT-5.4 at Low produces a minimal reasoning chain for an instant response, while at x-High it allocates significantly more thinking tokens for deeper analysis.

    Supported Models and Levels¶

    ModelLowMediumHighx-High
    GPT-5.4 (default)
    GPT-5.3-Codex (default)
    GPT-5.2-Codex (default)
    Claude Sonnet 4.6—
    Claude Opus 4.6—

    Unsupported models (no reasoning level selector):

    • GPT-4.1, GPT-4o, GPT-5 mini — non-reasoning models
    • Claude Haiku 4.5, Gemini 3 Flash — speed-optimized

    Enabling x-High

    x-High does not appear in the VS Code UI. Set it manually in settings.json:

    {
  "github.copilot.chat.responsesApiReasoningEffort": "xhigh"
}

    Cost: Reasoning Level Doesn't Matter¶

    This is the most important fact.

    Raising the reasoning level does not increase premium request consumption. Using GPT-5.2-Codex at Low costs the same 1 request as using it at x-High.

    Cost is determined by model selection, not reasoning level.

    ModelMultiplier
    GPT-4.1 / GPT-4o / GPT-5 mini0 (unlimited)
    Claude Haiku 4.5 / Gemini 3 Flash0.33
    Claude Sonnet 4.6 / GPT-5.2-Codex1
    Claude Opus 4.63

    This means the only downside to a higher reasoning level is increased latency. There is no cost reason to use Low.

    What Each Level Actually Does¶

    LevelBehaviorLatencyBest For
    LowMinimal reasoning. Instant responseFastestBoilerplate, syntax questions
    MediumBalanced (default)FastGeneral coding
    HighDeep reasoning chainSlightly slowerRefactoring, design decisions
    x-HighMaximum reasoning budgetSlowestSecurity audits, large migrations

    The Speed Difference Is Small¶

    In practice, the latency gap between High and Medium is a few seconds — barely noticeable. x-High isn't "dramatically slow" either, and the accuracy improvement on complex tasks is tangible.

    Low is genuinely faster, but response quality drops noticeably on certain tasks. Multi-file refactoring at Low, for example, can miss cross-file dependencies.

    Benchmarks: x-High Delivers¶

    OpenAI runs all official model benchmarks at x-High reasoning. While comparative data across levels isn't published, x-High clearly extracts peak performance.

    ModelSWE-Bench ProReasoning Level
    GPT-5.457.7%x-High
    GPT-5.3-Codex55.6%x-High

    GitHub's own blog reports that adopting advanced reasoning models increased latency by 16% but improved positive feedback rate by 6%.

    Does Low Have a Place?¶

    Yes, but it's limited.

    Low works well for:

    • Simple code completions and snippet generation
    • "What's the signature for this function?" type questions
    • Rapid feedback loops during prototyping
    • Auto-generating doc comments

    Avoid Low for:

    • Multi-file refactoring
    • Security review and vulnerability analysis
    • DB schema changes or framework migrations
    • Architecture decisions

    The rule is simple: tasks that don't require thinking → Low. Tasks that need thinking → High or above. But since cost is identical, default to High when in doubt.

    Recommended Configuration¶

    Run at High by default. Switch to x-High for demanding tasks.

    settings.json (recommended)
    {
  "github.copilot.chat.responsesApiReasoningEffort": "high"
}

    For maximum accuracy at all times:

    settings.json (max accuracy)
    {
  "github.copilot.chat.responsesApiReasoningEffort": "xhigh"
}

    Switching on the fly

    VS Code 1.113+ lets you switch levels from the model picker submenu with one click. Bump to High before a major refactoring session, drop to Medium for routine completions.

    Summary¶

    • Reasoning levels control how much the model "thinks." The model itself doesn't change
    • Premium request consumption is the same regardless of level — no cost reason to use Low
    • The speed difference is small; the accuracy difference is significant on complex tasks
    • Default to High, use x-High for demanding work. Low is for boilerplate only

    Many developers leave the default at Medium simply because it's the default. But if accuracy improves at no extra cost, there's no reason not to switch to High. It's one line in your settings.
