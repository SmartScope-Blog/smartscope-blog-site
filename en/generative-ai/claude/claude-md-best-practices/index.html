      📖 What is CLAUDE.md?¶

      CLAUDE.md is a configuration file that provides project-specific instructions to Claude Code (AI development assistant). By properly configuring this file, AI can understand your project's rules and conventions, providing more effective assistance.

      Basic Structure¶

      # CLAUDE.md

## Project Overview
Description of project purpose and structure

## Development Guidelines
- Coding standards
- Naming conventions
- Architecture patterns

## Workflows
Standard procedures and commands

## Checklists
Quality control checkpoints

      3. Development Guidelines¶

      ## Development Guidelines

### Code Style
- **Language**: TypeScript with strict mode
- **Linting**: ESLint with Airbnb config
- **Formatting**: Prettier with 2-space indentation
- **Naming**:
  - Components: PascalCase (e.g., `UserProfile`)
  - Functions: camelCase (e.g., `getUserData`)
  - Constants: UPPER_SNAKE_CASE (e.g., `API_BASE_URL`)
  - Files: kebab-case (e.g., `user-profile.tsx`)

### Git Conventions
- **Branches**: feature/*, bugfix/*, hotfix/*
- **Commits**: Conventional Commits format
  - feat: New feature
  - fix: Bug fix
  - docs: Documentation
  - style: Code style
  - refactor: Code refactoring
  - test: Test updates
  - chore: Build/tooling updates

### Testing Requirements
- Unit tests: Jest with 80% coverage minimum
- Integration tests: Cypress for critical paths
- Performance tests: Lighthouse score > 90

      🧠 Auto Memory¶

      Claude Code automatically saves context and decisions to ~/.claude/projects/<project>/memory/. Note that MEMORY.md (the index) only loads the first 200 lines. For details on how it works and how to disable it, see Command Reference > Auto Memory Control.

      📁 Rule Modularization with .claude/rules/¶

      Cramming all instructions and constraints (coding style, test strategy, deployment steps, etc.) into CLAUDE.md leads to heavy token consumption and unnecessary context bloat. The 2026 best practice is to split purpose-specific rule files into the .claude/rules/ directory and reference them from CLAUDE.md via @import or links as needed.

      .claude/rules/
├── code-style.md   # [TypeScript style] Referenced only when editing TS files
├── security.md     # [Auth & permissions] Referenced when editing security-related files
└── frontend/
    └── testing.md  # [Frontend test strategy] Referenced when writing test code

      This keeps CLAUDE.md focused on "core system knowledge that only humans know (tacit knowledge)" and enables smarter agent operation.

# CLAUDE.md - REST API Project

## Architecture Overview
      src/ ├── controllers/ # Request handlers ├── services/ # Business logic ├── models/ # Data models ├── middleware/ # Express middleware ├── utils/ # Helper functions └── config/ # Configuration files 
      ## API Development Guidelines

### Endpoint Naming
- Use nouns, not verbs
- Use plural for collections
- Use kebab-case for multi-word resources

Examples:
- GET /api/users
- GET /api/users/:id
- POST /api/users
- GET /api/user-profiles

### Response Format
```json
{
  "success": true,
  "data": {},
  "error": null,
  "metadata": {
    "timestamp": "2024-01-01T00:00:00Z",
    "version": "1.0.0"
  }
}

      // Always use this error format
throw new AppError('Error message', 400, 'ERROR_CODE');

      All endpoints must have: 1. Unit tests for business logic 2. Integration tests for API endpoints 3. Load tests for performance-critical endpoints

      • Rate limiting implemented
      • SQL injection prevention
      • XSS prevention
      • CORS properly configured
