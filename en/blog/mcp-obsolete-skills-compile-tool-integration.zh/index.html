    MCP的时代结束了？Skills编译时代的工具集成设计¶

    目标读者

    • 希望理清MCP与Skills区别的工程师
    • 希望在Agent设计中优化Token效率的实践者
    • 想了解「MCP已过时」这一说法真相的开发者

    本文要点¶

    • 「MCP已过时」这一说法究竟指的是什么
    • Skills（基于SKILL.md）的理念及其与MCP的差异
    • 上下文消耗在哪些环节产生
    • 最终应该采用哪种方案的判断标准

    先说结论：被质疑的不是MCP协议，而是使用模式¶

    首先明确一点。

    MCP协议本身并没有过时。被批评的是「将MCP服务器作为庞大的API字典公开，让LLM边探索边反复执行低级操作」这种使用模式。

    换言之，问题不在于「是否使用MCP」，而在于「如何使用MCP」。

    相关文章

    关于MCP与Skills的基本定位，请参阅「MCP vs Agent Skills论争的本质整理」。

    术语说明¶

    在进入正题之前，先简要整理本文使用的术语。

    术语含义
    MCP（Model Context Protocol）LLM调用外部工具的通用接口
    上下文LLM一次能处理的信息范围（对话历史＋指令＋数据）
    TokenLLM处理文本的单位。中文大约1个字≈1〜2个Token
    Skills将工作流程打包后传递给Agent的机制（由Anthropic提出）

    MCP基础：LLM调用外部工具的机制¶

    MCP是一种规范LLM调用外部工具时「通用协议」的协议。

    1. 客户端（IDE或聊天界面）连接到MCP服务器
    2. LLM调用服务器提供的工具（函数）
    3. 结果返回给LLM，并反映到对话中

    到这里很简单。问题出现在接下来的部分。

    想全面了解MCP？

    MCP的结构和用例全貌请参阅「MCP完全理解仪表盘」。

    「MCP导致上下文融化」到底发生了什么¶

    「上下文融化」这个说法经常被使用，但Token消耗分为两种类型。区分这一点很重要。

    探索Token（工具选择与流程探索）¶

    LLM在探索「该用哪个工具」「参数是什么」的过程中消耗的Token。

    • 应该使用哪个API？
    • 参数格式是什么？
    • 失败了，还有别的方法吗？

    这种「探索做法的对话」不断积累，会压缩上下文窗口。

    实际数据Token（处理对象的数据量）¶

    实际获取的数据（Slack日志、文档内容等）所消耗的Token。

    无论是MCP还是Skills，只要数据量大就必然产生。

    关键点

    当人们说「MCP很重」时，主要问题出在探索Token上。

    Slack集成示例：MCP容易出现什么问题¶

    以一个具体例子来说明：「收集特定Slack线程并生成会议纪要」。

    当MCP服务器公开了大量Slack低级API时，LLM往往会进行如下探索：

    1. 寻找线程定位方法（搜索？历史记录？URL指定？）
    2. 从候选中筛选的方法
    3. 用获取API提取正文
    4. 遇到速率限制或权限错误 → 探索替代路径
    5. 将获取的日志整理成会议纪要

    这种「探索＋试错」推高了探索Token的消耗。

    什么是Skills：工作单元的打包¶

    Skills是一种将工作流程预先打包后传递给Agent的方法。

    组成要素如下：

    • SKILL.md：描述流程、规则和输入规范的文件
    • 脚本：自动化定型处理的代码
    • 相关资源：模板和配置文件

    Skills的要点在于缩小LLM可见的「探索范围」。

    不是展示整个Slack API，而是只传递「生成会议纪要」这一任务的入口。这样就能减少探索Token。

    Skills不是万能的¶

    但是，引入Skills并不意味着探索为零。

    LLM终究要读取SKILL.md来做判断。如果SKILL.md写得模糊，或者出现意料之外的输入，探索仍然会发生。

    Skills发挥效果需要满足以下条件：

    • 工作可以标准化
    • 输入规范明确
    • 异常处理有限

    需要注意的是，Anthropic官方指南建议将SKILL.md正文控制在500行以内。Skills也有上下文管理的约束。

    MCP与Skills可以并用¶

    虽然到目前为止我们是对比着说明的，但MCP和Skills并非二选一。

    根据Anthropic的官方立场，MCP负责将LLM连接到外部服务，Skills负责教授工具的有效使用方法——两者被设计为组合使用。

    例如，通过Notion的MCP服务器建立连接，同时用Notion专用的Skills教LLM如何处理数据，这样的架构是可行的。

    MCP更占优势的场景¶

    那么，MCP在什么场景下更有效？答案是Token以外的因素占主导地位的情况。

    需要集中管控时¶

    审计日志、权限管理、速率限制等可以在服务器端统一管理。MCP架构重视安全性和受控访问，使组织能够严格管理AI助手的连接对象。

    需要在多个客户端共享同一集成时¶

    IDE Agent、聊天界面、批处理、其他团队的Bot都可以使用相同的连接和权限模型。

    探索本身有价值时¶

    在「每次提问形式不同」「目的是跨领域搜索」等场景下，固定为Skills反而会失去灵活性。

    PoC阶段变化较大时¶

    在需求尚未确定的阶段就精心构建Skills，会导致大量返工。

    MCP集中化的权衡¶

    将集成集中到MCP服务器可以获得单一控制点，但也伴随以下风险：

    • 服务器故障影响所有Agent
    • API更新波及所有客户端
    • 服务器管理员成为瓶颈

    另一方面，Skills分散化也存在「技能散落重复」「审计困难」「属人化」等问题。

    这是集中与分散的权衡，没有绝对正确的答案。

    实现模式详解

    MCP服务器的具体实现模式请参阅「MCP Server实现模式」。

    判断标准：检查清单¶

    Skills优先的典型场景¶

    • 以每周1次以上的频率执行同类工作
    • 规范稳定
    • 失败成本高，需要可复现性
    • 可在本地或CI中完成

    MCP优先的典型场景¶

    • 每次提问形式不同（探索有价值）
    • 需要在多个团队、多个客户端间共享
    • 需要在一处集中管控审计、权限和密钥
    • PoC阶段需求流动性大

    总结¶

    「MCP已过时」这一说法的本质，不是对协议的批评，而是对使用模式的批评。

    • MCP擅长「连接、共享、管控」
    • Skills擅长「固化重复、减少探索」
    • 两者可以并用，根据目的灵活组合

    选择哪种方案，不仅要考虑Token效率，还需要综合管控、共享、应变能力等因素来判断。

    补充：Code Execution with MCP¶

    本文未涉及，但Anthropic在2025年提出了将MCP与代码执行结合的新模式。

    在该方式中，Agent将MCP服务器作为代码API使用，按需加载工具，在执行环境内处理数据后再返回给模型。官方测试报告显示，某些案例实现了从15万Token减少到2,000 Token的效果（减少98.7%）。

    对于涉及大量MCP服务器（10个以上）的复杂工作流，或需要处理大量中间数据的场景，这种方式尤为有效。

    详情请参阅Anthropic工程博客「Code Execution with MCP」。
