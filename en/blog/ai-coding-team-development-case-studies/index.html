    複数人でAIコーディング、何を揃えるか ― 3つの公開事例から読む暫定解¶

    対象 / ポイント

    対象: Claude Code や GitHub Copilot を個人では使い始めているが、 チーム導入や共同開発への展開に悩んでいるテックリード、EM、開発者。

    ポイント: - 複数人でのAIコーディングで効くのは、個人のプロンプト力より共有資産の設計 - 公開事例で共通していたのは、指示ファイル、規約、サンプル実装、レビュー観点の整理 - 本記事は正解の提示ではなく、公開事例から読める暫定解の整理として書く

    Claude Code や GitHub Copilot の活用事例は増えた。
    ただし多くは、個人開発か、強いエンジニア1人がAIを使いこなす話に寄りやすい。
    実際の現場で問題になるのは、同じプロダクトを複数人で開発するときの運用である。

    個人のプロンプト術ではなく、チームとして何を揃えるか。
    本記事では、複数人で同一リポジトリや同一案件を運用している公開事例だけを絞って見ていく。

    対象事例の選定条件¶

    今回は、以下を満たす事例に絞った。

    • 同一プロダクト、案件、またはリポジトリを複数人で開発している
    • 実際の運用ルールや役割分担が公開情報から読み取れる

    全社PoC、AIチャンピオン制度、導入効果アンケートは今回は対象外としている。
    本記事で確認した公開事例の範囲では、この条件を満たす事例はまだ多くない。

    事例1: NTTデータ ― 3人チームで顧客案件を半年運用¶

    NTTデータは、顧客向け案件を3人チームで半年にわたりAI駆動開発で進めた事例を公開している。1
    注目すべきは、プロンプトや規約をリポジトリ側に集約した点だ。.github 配下にプロンプト、 コーディング規約、システム構造定義、タスク定義が配置されている。1

    個人が思いつきで書いたプロンプトを投げる運用は禁止されており、修正が必要なら共有プロンプトの更新か、 レビュー指摘票を経由する。1
    結果として、AIが生成した成果物に対して人間レビューを通すという、通常の開発プロセスに近い形で整えられている。

    事例2: mediba ― Copilot中心の開発フロー¶

    medibaは、1つの技術刷新プロジェクトで GitHub Copilot を中心に据えた開発フローを構築している。2
    要件定義、設計、実装、テストの各工程で Copilot を使い分けているが、重要なのはその前段である。
    人間が先にサンプル実装とコーディング規約を整備し、そこからAI利用に入っている。2

    運用面では、*.prompt.md と *.instructions.md を Git で管理している。2
    プロンプトや指示がリポジトリ内に可視化されるため、誰が何をAIに依頼しているかが追える。
    AI出力の属人化を抑える仕組みが、ファイル構成として埋め込まれている。

    事例3: 5人チームのNext.js開発¶

    5人チームで Next.js の Web アプリケーションをAI駆動で開発している事例では、最初の課題は速度ではなく、 コード品質のばらつきだった。3
    同じ要件でも、書き手によってスタイルや抽象化の粒度が揃わない問題がある。

    対応として、過去のレビュー指摘を .github/copilot-instructions.md に集約した。3
    Next.js / React のベストプラクティス、API方針、エラーハンドリング、セキュリティ、性能観点が 明文化されている。3
    結果として、人間レビューの重心が「規約に合っているか」から「ロジックが正しいか」へ移ったと 報告されている。3

    3事例に共通していた運用パターン¶

    事例ごとの色は違うが、共通する構造が見える。

    ルールの置き場をリポジトリに置く¶

    3事例とも、運用ルールは Wiki や口頭ではなく、リポジトリ内に置かれている。
    GitHub 側にも、リポジトリ全体に効く instructions、パス単位の instructions、AGENTS.md といった 仕組みが公式に用意されている。4
    コードとルールを同じ場所で管理すると、ルール変更そのものがレビュー対象になる。

    AIの前に規約とサンプル実装を用意する¶

    3事例とも、AIに書かせる前に人間が規約やサンプル実装を先に置いている。
    AIは白紙から全部決める存在ではなく、既に定義された型に沿って生成する存在として扱われている。
    この順序が逆になると、生成物ごとにスタイルが割れやすい。

    個人の自由入力に寄せすぎない¶

    NTTデータは自由プロンプトを明示的に禁止し、1 mediba は prompt / instructions を Git 管理し、2 5人チーム事例は instructions ファイルを共有している。3
    個人が思いつきでAIに話しかける運用は、3事例のいずれも主軸ではない。
    プロンプトや指示は、チームの共有資産として扱われている。

    レビューを規約確認からロジック確認へ寄せる¶

    5人チーム事例で最も明示的だが、3 他の2件でも同じ方向が読める。
    規約違反や命名の崩れは AI 側の制約で抑え、人間は設計判断やロジック妥当性に集中する。
    AI導入でレビューが消えるのではなく、レビューの内容が変わるという整理のほうが実態に近い。

    まとめ¶

    3事例から見えるのは、AIコーディングをチームで回すとき、効いているのは個人のプロンプト力ではなく、 共有資産の設計だという点である。
    現時点のパターンは、「チームの合意を機械可読な形で置く」という一点に収束している。

    ただし、公開事例はまだ限定的で、これをベストプラクティスと言い切れる段階ではない。
    本記事はあくまで、公開事例から観測できる暫定解としての整理である。
