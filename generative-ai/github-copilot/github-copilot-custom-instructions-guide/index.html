mkdir -p .github && touch .github/copilot-instructions.md

# 2. 最小限のインストラクションを記述
cat << 'EOF' > .github/copilot-instructions.md
# プロジェクト開発ガイドライン

# 3. コミットして共有
git add .github/copilot-instructions.md
git commit -m "Add Copilot custom instructions"

      🗺️ インストラクション体系の全体像¶

      GitHub Copilotのインストラクションは4つの階層 + 2つの拡張仕組みで構成されています。本記事はこの中の「Repository-wide」と「Path-specific」を中心に解説します。

      graph TB
    P["<b>1. Personal</b>（個人設定）<br>VS Code / GitHub.com のユーザー設定"]
    O["<b>2. Organization</b>（組織ポリシー）<br>Business/Enterprise の管理者設定"]
    R["<b>3. Repository-wide</b>（リポジトリ全体）<br><code>.github/copilot-instructions.md</code>"]:::focus
    PA["<b>4. Path-specific</b>（パス別）<br><code>.github/instructions/*.instructions.md + applyTo</code>"]:::focus

    P --> O --> R --> PA

    SK["<b>Agent Skills</b>（オンデマンド起動）<br><code>.github/skills/&lt;name&gt;/SKILL.md</code>"]:::ext

      本記事の対象範囲

      上図で青色ハイライトされている 3. Repository-wide と 4. Path-specific を中心に解説します。 拡張仕組みの詳細: Agent Skillsガイド ・ AGENTS.md統一管理ガイド

      仕組み適用タイミング典型的な用途
      copilot-instructions.md常時（全リクエスト）コーディング規約、技術スタック
      .instructions.md + applyToファイルパスが一致したときフロント/バックエンド別ルール
      Agent Skillsエージェントが必要と判断したときTerraformレビュー手順、リリース手順
      AGENTS.mdエージェント起動時プロジェクト全体の方針（マルチツール共通）

      使い分けの原則

      • 常に適用したいルール → copilot-instructions.md / .instructions.md
      • 特定タスクでだけ必要な手順 → Agent Skills
      • 複数AIツールで共通化したい → AGENTS.md
      • 詳細な比較: applyToパターンガイド

      📖 GitHub Copilotカスタムインストラクションとは¶

      GitHub Copilotカスタムインストラクションは、2024年に本格的に実用化された機能で、プロジェクト固有の開発方針やコーディング規約をCopilotに事前に伝える仕組みです。これにより、毎回詳細な指示を与えることなく、プロジェクトに適したコード生成や提案を受けられます。

      指示ファイルは"AIのREADME"だ

      2025-2026年の主要アップデート（クリックで展開）

      機能追加:

      • Agent Mode (2025年2月発表・2025年4月全ユーザー展開) - マルチファイル編集・テスト実行・検証を自動化
      • 複数.instructions.md対応 (2025年7月) - YAML Frontmatterでパス別インストラクション
      • Copilot Spaces (2025年9月24日GA) - コード・ドキュメント・ノートを統合管理
      • Agent-Specific Instructions (2025年11月) - excludeAgentプロパティで特定エージェントへの適用制御
      • Code Review統合 - コードレビューでもcopilot-instructions.md対応

      モデル:

      • 最新対応モデル: Claude Opus 4.6（2026年2月5日GA、1Mトークン）、GPT-5.1シリーズ、Gemini 3 Pro — 対応モデル一覧
      • 2026年2月17日に Claude 3系・GPT-4系の一部が提供終了予定 — GitHub Changelog

      料金・プラン: GitHub Copilot総合ガイドを参照

      主な機能¶

      • .github/copilot-instructions.md によるリポジトリレベルの統一ルール
      • .instructions.md + applyTo によるファイル別適用
      • VS Code、Visual Studio、JetBrains IDEs、Neovim、GitHub.com、GitHub Mobile、GitHub CLIで利用可能
      • Copilot Code Reviewでも同じインストラクションを活用

      🎯 コンテキストサイズとファイル設計のベストプラクティス¶

      関連記事

      ファイルサイズの制約と推奨事項¶

      GitHub Copilotは新AIモデル（GPT-5.1、Claude Opus 4.6、Gemini 3 Pro）により拡張されたコンテキストウィンドウを持ちますが、カスタムインストラクションファイル自体は簡潔に保つことが重要です。

      絞るほど精度が上がる

      ファイルサイズに関する重要な指針

      カスタムインストラクションファイルに大量のコンテキストを直接記述することは推奨されません。代わりに、外部ドキュメントへの参照パターンを活用することで、効率的な運用が可能になります。

      推奨されるアプローチ¶

      # ❌ 非推奨: 大量のコンテキストを直接記述
コーディング規約：
1. 変数名は camelCase を使用
2. 関数名は動詞から始める
3. クラス名は PascalCase を使用
4. インデントはスペース2文字
5. 文字列リテラルはダブルクォート
6. セミコロンは省略しない
... (大量の詳細規約が続く)

# ✅ 推奨: 参照パターンを活用
コーディング規約は [coding-standards.md](/generative-ai/github-copilot/docs/coding-standards/) を参照してください。
デバッグ手順は [debugging-guide.md](/generative-ai/github-copilot/docs/debugging-guide/) に従って実施してください。
API設計方針は [api-design.md](/generative-ai/github-copilot/docs/api-design/) を確認してください。

      AGENTS.md について

      AGENTS.md は、Linux Foundation / Agentic AI Foundation が策定したオープン標準で、GitHub Copilot・Claude Code・Cursor・Gemini CLI など複数のAIツールで共通利用できるプロジェクト命令ファイルです。

      VS Code での有効化: 

      {
  "chat.useAgentsMdFile": true,
  "chat.useNestedAgentsMdFiles": true
}

      copilot-instructions.md との使い分け:

      • copilot-instructions.md: GitHub Copilot専用、常時適用
      • AGENTS.md: マルチツール共通、エージェント起動時に参照

      → 詳細: AGENTS.mdクロスツール統一管理ガイド

      Agent Skills について

      Agent Skills は、SKILL.md + 補助リソースをフォルダにまとめた「手順知識パッケージ」です。.github/skills/ に配置すると、エージェントが必要なときだけ読み込んで作業精度を上げます。

      copilot-instructions.md / .instructions.md が「常時適用のルール」なのに対し、Skills は「条件一致時にオンデマンド起動する専門能力」という位置づけです。

      → 詳細: GitHub Copilot Agent Skillsガイド

      Step 3: YAML Frontmatter活用（2026年2月版・互換性重視）¶

      VS Code と GitHub.com での違い

      • VS Code: name / description / applyTo を UI表示・選択性のメタとして活用可能
      • GitHub.com: 主に applyTo と excludeAgent を参照
      • applyTo の複数指定はカンマ区切り推奨（GitHub Docs準拠）

      frontend.instructions.md の例: 

      ---
# VS Code では name/description も活用可能（任意）
name: "Frontend instructions"
description: "React/TypeScript の実装ルール（UI/状態管理/命名など）"

# 適用範囲（GitHub Docs の説明に寄せてカンマ区切り推奨）
applyTo: "src/components/**/*.ts,src/components/**/*.tsx"

# 任意: 特定のエージェントには適用しない（例）
# excludeAgent: "code-review"
---

# フロントエンド開発指示

## React コンポーネント作成時
- TypeScript strict モードを使用
- React Hooks の依存配列を必ず指定
- コンポーネント名は PascalCase を使用

## スタイリング
- Tailwind CSS を使用（inline styles は避ける）
- レスポンシブ対応を必ず実装

      ---
# VS Code では name/description も活用可能（任意）

# excludeAgent: "code-review"
---

# バックエンド開発指示

## API エンドポイント作成時
- Express Router を使用
- エラーハンドリングを必ず実装
- OpenAPI 3.0 仕様に準拠

      applyTo の指定方法

      • glob パターンで、複数指定はカンマ区切り
      • excludeAgent で特定エージェント（coding-agent、code-review など）への適用を除外可能

      メインインストラクションファイルの構成例（2025年ベストプラクティス）¶

      # プロジェクト開発ガイドライン

## 🏗️ 技術スタック
- **フロントエンド**: Next.js 14 + TypeScript + Tailwind CSS
- **バックエンド**: Node.js + Express + Prisma ORM
- **データベース**: PostgreSQL
- **テスト**: Jest + React Testing Library
- **パッケージマネージャー**: pnpm（npm/yarn使用禁止）

## 📝 コーディング規約（重要）
- JavaScript/TypeScriptは **ダブルクォート** + **タブインデント**
- 変数・関数名: camelCase
- コンポーネント名: PascalCase
- 定数: UPPER_SNAKE_CASE

cat << EOF >> .github/copilot-instructions.md

## コーディング規約
- JavaScript/TypeScriptはダブルクォート使用
- インデントはタブまたはスペース2文字
- 変数名: camelCase、コンポーネント名: PascalCase

## 必須実装事項
- TypeScript strict モード使用
- エラーハンドリング必須実装
- テストコード作成必須
EOF

# 3. Git追加・コミット
git add .github/copilot-instructions.md
git commit -m "Add Copilot custom instructions"

      ⚙️ VS Code設定とワークフロー統合¶

      settings.json での追加設定（2026年2月版）¶

      deprecated設定に注意

      github.copilot.chat.codeGeneration.instructions は VS Code Docs で deprecated 扱いです。 基本は instruction files を使うことを推奨します。

      推奨の記述（最小構成）: 

      {
  // .github/copilot-instructions.md を自動で取り込む
  "github.copilot.chat.codeGeneration.useInstructionFiles": true,

  // *.instructions.md の探索場所（デフォルトもこれ）
  "chat.instructionsFilesLocations": {
    ".github/instructions": true
  }

  // ※ github.copilot.chat.codeGeneration.instructions は deprecated（原則使わない）
}

      デフォルト設定

      chat.instructionsFilesLocations のデフォルトは { ".github/instructions": true } です（VS Code設定リファレンス）。 instruction files が効かない時の切り分け（場所・設定）も VS Code Docs にまとまっています。

      🎯 まとめ：効果的なカスタムインストラクション戦略¶

      重要なポイント¶

      1. 簡潔性の維持: カスタムインストラクションファイル自体は簡潔に保ち、詳細は外部ドキュメントで管理
      2. 参照パターンの活用: [file](path)や#file:path記法で効率的にドキュメント参照
      3. 人間とAI両方への最適化: 新人エンジニアが読んでも理解できる明確さ
      4. 段階的詳細レベル: 即座に必要な情報から詳細ガイドラインまで階層化
      5. 継続的改善: 定期的なレビューと更新でプロジェクトの変化に対応

      実装優先度¶

      1. 基本セットアップ: .github/copilot-instructions.mdの作成
      2. コアドキュメント: コーディング規約、デバッグガイドの整備
      3. 参照パターン: 外部ドキュメントへの効果的な参照設計
      4. チーム導入: 設定共有と使用方法の教育
      5. 継続改善: 定期レビューサイクルの確立

      GitHub Copilotカスタムインストラクションは、単なる設定ファイルではなく、チーム開発の知識基盤を体系化する重要なツールです。適切に設計・運用することで、AI支援による開発効率化と品質向上を同時に実現できます。

      ❓ よくある質問 (FAQ)¶

      Q: copilot-instructions.md はどの IDE からでも共有できますか？

      はい、GitHub リポジトリの .github/ ディレクトリに配置するため、VS Code、Visual Studio、JetBrains IDEs など、GitHub Copilot がサポートする主要な IDE で共通して利用できます。

      Q: .instructions.md はどの Copilot 機能で効きますか？

      Copilot Chat および Agent Mode で有効です。インラインのコード補完（Ghost Text）には直接適用されませんが、Chat を通じたコード生成には強く影響します。

      Q: 個人設定とリポジトリ設定はどちらが優先されますか？

      基本的には両方のコンテキストが考慮されますが、競合する場合はユーザー個人の Custom Instructions 設定が優先される傾向にあります。ただし、リポジトリ固有のルールを強制したい場合は、チーム内で設定の同期を推奨します。
