    GitHub Copilot 完全ガイド

    GitHub Copilotが使用量課金へ移行 - 2026年6月から始まるAI Credits体系の全貌¶

    対象 / ポイント

    対象: GitHub Copilot Pro / Pro+ / Business / Enterpriseを利用中の個人・組織管理者。

    ポイント:

    • 2026年6月1日からCopilot全プランはPremium Request UnitsからGitHub AI Creditsへ移行する1。
    • 月額基本価格は据え置きだが、agenticセッションの実コストと年契約Pro / Pro+のmodel multiplier上昇が負担増の焦点になる2。
    • 個人は含み枠、組織はpooled creditsと予算制御を、5月開始のpreview billで実測する必要がある1。

    2026年6月1日、GitHub Copilotの課金単位が変わる。 これまでのPremium Request Units（PRU）は廃止され、入力・出力・キャッシュトークンをGitHub AI Creditsへ換算する方式に移行する1。

    この記事の問いはシンプルだ。 Copilot利用者は、6月1日までに何を測り、どの予算ルールを決めるべきか。

    PRUからAI Creditsへ何が変わるのか¶

    新方式の本質は、リクエスト数ではなくトークン消費量が請求単位になることだ。 短いチャット質問と、数時間の自律コーディングセッションを同じPRUで扱う設計は、agentic利用の増加に耐えにくくなっていた1。

    GitHub AI Creditsでは、入力トークン、出力トークン、キャッシュされた文脈がモデル別の公開単価で計算される。 換算レートは固定で、1 AI credit = 0.01 USDである2。 code completionsとNext Edit suggestionsはAI Creditsを消費せず、有料プランでは引き続き無制限に含まれる2。

    大きな違いは、PRU枯渇時の低コストモデルへの自動フォールバックがなくなる点だ。 新方式では、利用継続は残クレジット、追加利用予算、管理者ポリシーで決まる1。

    ここから先は、個人も組織も「何回使ったか」ではなく「どのモデルで、どれだけ長い文脈を処理したか」を見る運用になる。

    プラン別の含み枠と超過時の挙動¶

    月額基本価格は据え置きだが、含まれるAI Creditsの量がプランごとに変わる。 個人プランではProが月1,000 credits、Pro+が月3,900 creditsで、これはそれぞれ10 USD、39 USDの月額価格に対応する3。

    プラン月額月次含みCreditsプロモ枠超過時の基本挙動
    Copilot Pro10 USD1,000なし追加利用予算を設定して継続、または次回リセット待ち
    Copilot Pro+39 USD3,900なし同上
    Copilot Business19 USD / seat1,900 / user3,000 / user組織プールと追加利用ポリシーで制御
    Copilot Enterprise39 USD / seat3,900 / user7,000 / user組織プールと追加利用ポリシーで制御

    BusinessとEnterpriseでは、各ユーザーの含み枠がbilling entity単位でプール化される4。 たとえばBusiness 100シートなら、1,900 credits x 100で月190,000 creditsの共有プールになる。 ヘビーユーザーが余剰を使える一方、ライトユーザーの未使用枠が死蔵されにくい。

    既存Business / Enterprise顧客には、 2026年6月1日から9月1日までの3か月間、プロモーション枠が付く4。 Businessは3,000 credits / user、Enterpriseは7,000 credits / userになる。 この期間は本番運用の猶予ではなく、組織内の利用偏在を測るための実測期間と見るべきだ。

    主要モデルの単価レンジ¶

    モデル選択は、そのまま支出の差になる。 GitHub Docsが公開している2026年6月1日以降の単価は、1Mトークンあたり次の通りだ2。

    GPT-5 miniやGrok Code Fast 1のようなLightweight層は、短い修正や説明には低コストで使いやすい。 一方、GPT-5.5やClaude Opus 4.7のようなPowerful層は、出力単価だけでLightweight層の10倍以上になる場合がある。

    Anthropicモデルでは、cached inputとは別にcache write単価も発生する2。 Sonnet 4.6は3.75 USD / 1M、Opus 4.7は6.25 USD / 1Mのcache writeがあるため、長文コンテキストを頻繁に更新するワークフローでは見落としやすい。

    agenticセッションが重くなる理由¶

    agenticセッションでは、1つの依頼が複数回のモデル呼び出しに分解される。 ユーザーが「この機能を直して」と依頼すると、エージェントは複数ファイルを読み、ツールを呼び、直前文脈を再投入し、実装と検証を繰り返す。

    この動きは便利だが、課金上は入力と出力が積み上がる。 GitHub Docsも、agent modeやCopilot cloud agentのような機能は、単発のチャット質問より大きく消費し得ると説明している3。

    実務上の最適化は3つに絞られる。

    • モデル区分を分ける: 軽微な修正、説明、ファイル探索はLightweight層で済ませる。
    • 探索範囲を明示する: 対象ディレクトリ、変更禁止ファイル、完了条件を先に渡す。
    • 再利用文脈を固定する: 反復利用するルールや前提を安定した場所に置き、無駄な再投入を減らす。

    Copilot code reviewだけは、さらに注意がいる。 トークン消費に加えて、GitHub-hosted runners上のGitHub Actions minutesも消費する2。 self-hosted runnersならActions minutesは消費しないが、運用コストは自前インフラ側に移る。

    個人ユーザーが5月に確認すべきこと¶

    個人ユーザーの最初の作業は、5月に始まるpreview billで自分の消費傾向を見ることだ。 GitHubは、6月1日の移行前にBilling Overviewから projected costs を確認できるpreview bill experienceを提供すると説明している1。

    特に年契約Pro / Pro+のユーザーは、月額ユーザーと扱いが違う。 既存年契約は満了までPRU方式が続くが、2026年6月1日からannual subscribers向けのmodel multiplierが引き上げられる2。

    Claude Opus 4.77.527
    Claude Sonnet 4.619

    年契約満了時はCopilot Freeへ移行し、必要に応じて有料月額プランへアップグレードする流れになる1。 満了前に月額へ移行する場合は、残存価値に応じたprorated creditsが付与される1。

    Powerful層を多用している人ほど、年契約のまま高いmultiplierを受け入れるか、月額へ移るかの損益分岐を計算する必要がある。

    組織管理者は予算ルールを先に決める¶

    組織の焦点は、追加利用を許すか、どこで止めるかだ。 Business / Enterpriseでは、プール枠が尽きたあとも追加利用を許可すれば公開単価で継続利用でき、許可しなければ次の請求サイクルまでブロックされる4。

    予算制御は複数階層で設計する必要がある。 GitHub Blogはenterprise、cost center、user levelsのbudget controlsを導入すると説明している1。 組織運用では、少なくとも次の3点を先に決めておきたい。

    • 全体上限: enterpriseまたはbilling entityとして、月次の最大支出を決める。
    • 部門配賦: cost centerで開発部門、研究開発、情シスなどに分けて見る。
    • 個人上限: 特定ユーザーの暴走を止める一方、緊急時の解除フローを用意する。

    重要なのは、ユーザー単位で止めたときに低コストモデルへ自動退避しないことだ。 アクセス制御は明確になるが、設定を誤ると開発作業そのものが止まる。

    予算は「締める」ためだけの仕組みではない。 preview billの実測値を使い、どのチームがagentic利用で成果を出し、どこが高コストな探索を繰り返しているかを見分けるための観測点になる。

    移行前チェックリスト¶

    5月のpreview billは、移行前の最後の設計材料になる。 個人も組織も、6月1日を待たずに以下を確認しておきたい。

    1. preview billで主要ユーザーの月次予測コストを確認する。
    2. Copilot CLI、cloud agent、code reviewの利用頻度とモデル選択を棚卸しする。
    3. Powerful層を使うタスクとLightweight層で足りるタスクを分ける。
    4. 組織では追加利用の許可、上限、例外申請フローを決める。
    5. 年契約Pro / Pro+では、multiplier上昇後の実質コストを再計算する。

    このチェックは、コスト削減だけが目的ではない。 agentic開発を止めずに使うために、どの利用を投資として許容するかを決める作業でもある。

    まとめ¶

    今回の変更は、Copilotの値付けだけの話ではない。 AIコーディングツールが、固定価格の補助金モデルから、実消費に近い価格モデルへ移る転換点だ。

    月額価格が変わらなくても、実消費が増えれば支払いは増える。 逆に、モデル選択、スコープ指定、キャッシュ前提の文脈設計ができる組織なら、含み枠内で十分に運用できる余地もある。

    6月1日以降、Copilotのコスト管理は経理だけの仕事ではなくなる。 開発チームが「どのタスクにどのモデルを使うか」を設計できるかどうかが、agentic開発の生産性と予算を同時に左右する。
