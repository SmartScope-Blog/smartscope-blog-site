    GitHub Copilotの学習データポリシー変更、個人開発者が確認すべき設定¶

    GitHub Copilot 完全ガイド

    対象 / ポイント

    対象: GitHub Copilot Free / Pro / Pro+ を個人で使っている開発者

    ポイント:

    • 2026年4月24日から、対象プランのinteraction dataは既定でAIモデル改善に使われる
    • Business / Enterprise と paid organization のリポジトリ由来データは除外される
    • 個人設定の Allow GitHub to use my data for AI model training をDisabledにする

    2026年4月24日、GitHub Copilotのデータ利用ポリシー変更が発効しました。 対象はCopilot Free / Pro / Pro+の個人ユーザーです。 何もしない場合、Copilotとのやりとりの一部がAIモデルのトレーニングと改善に使われます1。

    この記事の問いはシンプルです。 個人開発者は、どの設定を確認し、どのコード利用を避けるべきか。

    • 発効済みの変更 告知ではなく、すでに適用中の設定として確認する。
    • 対象プラン Free / Pro / Pro+ とBusiness / Enterpriseを分ける。
    • private repoの扱い 保存中のリポジトリ全体と、Copilotに送った断片を区別する。
    • 設定手順 Copilot settingsで学習利用を無効化する。

    2026年4月24日から何が変わったか¶

    変更点は、Copilotの入力・出力・コードスニペット・関連コンテキストが、対象プランでは既定でモデル改善に使われるようになったことです。 GitHubは2026年3月25日にこの方針を告知し、30日後の2026年4月24日に発効すると説明しました1。

    重要なのは、これは利用可否の設定ではなく、学習利用の設定だという点です。 Copilotを使い続けるかどうかとは別に、自分のinteraction dataをモデル改善へ使わせるかを選びます。 したがって、まず確認する場所はIDEではなくGitHubアカウント側のCopilot設定です。

    ここでいうinteraction dataは、単なる利用回数ログより広い概念です。 コード生成時にCopilotへ送られる周辺文脈や、受け入れた出力も対象に含まれます。 「エディタで見えている作業中コードの一部が、サービス実行のために送信される」という前提で考える必要があります。

    対象プランと対象外プラン¶

    今回の変更は、個人向けプランに絞って確認すると判断しやすくなります。 Business / Enterpriseで付与されたCopilot seatを使っている場合、GitHub Docsは顧客データをAIモデルのトレーニングに使わないと説明しています2。

    利用形態学習利用の扱い確認ポイント
    Copilot Free対象個人設定で無効化できる
    Copilot Pro対象個人設定で無効化できる
    Copilot Pro+対象個人設定で無効化できる
    Copilot Business / Enterprise対象外DPAにより保護され、設定項目が表示されない場合がある
    学生・教師の無料Pro対象外GitHub FAQで除外と説明されている

    個人アカウントでProを契約している人ほど、設定確認の優先度が高いです。 無料枠だけで試している人も、private repoや個人開発の未公開コードを触るなら同じ確認が必要です。

    収集対象になるinteraction data¶

    GitHubの説明では、設定が有効な個人ユーザーについて、Copilot利用時の複数のデータがモデル改善に使われ得ます1。 「promptだけ」でも「完成コードだけ」でもありません。

    • Copilotに送った入力: プロンプト、コードスニペット、モデルに示された周辺文脈
    • Copilotからの出力: 受け入れた提案、修正した提案、チャットやインライン提案
    • 作業文脈: カーソル周辺コード、コメント、ドキュメント、ファイル名、リポジトリ構造
    • 利用行動: Copilot機能とのやりとり、提案へのフィードバック

    GitHubは、第三者のAIモデル提供者に独立した学習目的で入力・出力を共有しないとも説明しています3。 一方で、GitHub affiliates、主にMicrosoftを含む企業グループ内での利用可能性は明記されています。 個人開発者にとっては「外部AI企業に売られるか」だけでなく、「GitHub/Microsoft側の学習に使われるか」が判断軸になります。

    個人開発者が確認すべき設定手順¶

    確認する設定は、GitHubのWeb側にあります。 Docsでは、GitHub右上のプロフィール画像からCopilot settingsに入り、Allow GitHub to use my data for AI model training をDisabledにする手順が示されています2。

    1. GitHubにログインする
    2. github.com/settings/copilot を開く
    3. Copilot settingsのPrivacyまたはModel training関連項目を確認する
    4. Allow GitHub to use my data for AI model training を Disabled にする
    5. ページを再読み込みし、設定がDisabledのままか確認する

    以前、prompt and suggestion collectionを無効化していた場合、その選択は引き継がれるとGitHub FAQは説明しています4。 ただし、UI表示名や配置は変わり得ます。 設定名が見つからない場合は、Business / Enterprise seatでログインしていないかを先に確認してください。

    プライベートリポジトリと業務コードの注意点¶

    private repo全体が保存状態のまま吸い上げられる、という説明ではありません。 GitHubは、private repository at restの内容をトレーニングに使わない一方、Copilot使用中にモデルへ送られたコード断片は対象になり得ると説明しています4。

    この違いが実務上の落とし穴です。 「リポジトリがprivateだから安全」ではなく、Copilotに渡った断片が対象になるかを見ます。 秘密情報、未公開アルゴリズム、顧客固有ロジックを含む作業では、設定をDisabledにしてからCopilotを使うのが現実的です。

    もう1つの重要な補正があります。 GitHub FAQは、paid organizationのリポジトリ内容について、個人プランで作業していてもモデル学習しないと説明しています4。 雇用先や有料組織のリポジトリで作業する場合と、個人所有のprivate repoで作業する場合を混同しないでください。

    反発点と実務上の判断¶

    反発の中心は、オプトアウト方式です。 GitHubはユーザーに設定変更の選択肢を用意した一方、明示的に拒否しなければ対象になる設計を採りました。 Community FAQにも、private repo、企業コード、同意方式に関する質問が集まっています4。

    個人開発者の判断は、次の3分類にすると迷いにくくなります。

    作業内容推奨判断理由
    公開予定のOSSや学習用コード任意公開前提ならリスクは相対的に低い
    個人のprivate repoや未公開サービスDisabled推奨Copilotに送った断片が学習対象になり得る
    業務コード・顧客コード組織ポリシー優先paid organization除外の確認と社内ルール確認が必要

    この設定は、技術力やCopilotへの評価とは別問題です。 AI補助を使うことと、作業中データをモデル改善へ提供することは分けて判断できます。

    まとめ¶

    2026年4月24日以降、Copilot Free / Pro / Pro+の個人ユーザーは、学習データ設定を一度確認しておくべきです。 特にprivate repoでCopilotを使う人は、Allow GitHub to use my data for AI model training がDisabledかを確認してください。

    新しい示唆は、Copilotのリスク管理が「ツール導入時の一度きりの判断」ではなくなったことです。 AIエージェントがIDE、CLI、GitHub上のissueやPRまで広がるほど、設定は機能単位で増えます。 今後は、補完精度だけでなく、どの画面のどのデータ利用設定を監査するかまで開発環境の運用項目になります。
